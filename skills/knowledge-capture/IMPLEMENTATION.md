# Knowledge Capture - 实现说明

本文档说明了 Knowledge Capture Skill 的实现细节、设计决策和测试方法。

## 📁 文件结构

```
knowledge-capture/
├── .gitignore                # Git 忽略规则
├── README.md                 # 用户文档（详细使用说明）
├── QUICKSTART.md            # 快速开始指南
├── IMPLEMENTATION.md        # 本文件：实现说明
├── SKILL.md                 # 核心 Skill 文件（工作流逻辑）
└── references/              # 参考资料目录
    ├── categories.md        # 分类映射规则
    └── templates.md         # 文档模板定义
```

## 🎯 实现方案

### 技术选型：Skill（而非 MCP Server）

**选择理由：**

1. **需求简单明确**：工作流是"分析 → 生成 → 推送"，不需要复杂的 API
2. **快速实现**：单个 SKILL.md 文件即可完成所有逻辑
3. **易于维护**：纯文本配置，用户可以轻松自定义
4. **灵活交互**：可以在流程中多次与用户交互，动态调整
5. **上下文感知**：可以访问对话历史，分析内容类型

**与 MCP Server 对比：**

| 特性 | Skill | MCP Server |
|------|-------|------------|
| 开发复杂度 | 低（Markdown + Bash） | 高（完整项目） |
| 集成难度 | 简单（直接放到目录） | 复杂（配置 server） |
| 维护成本 | 低 | 中高 |
| 功能扩展性 | 中等 | 强 |
| 适用场景 | 工作流自动化 | 复杂 API 集成 |

MCP Server 更适合需要持久化状态、复杂认证、大量 API 端点的场景。

### 依赖管理

**必需依赖：**
- Git Action Skill（处理所有 Git 操作）
  - 如果没有，使用 Bash 工具执行 Git 命令（已在 SKILL.md 中实现）

**可选依赖：**
- 图床上传 Skill（上传图片到 GitHub 图床）

## 🔄 工作流程详解

### 第一步：依赖检查

- 检查 Git Action Skill（必需）
- 检查图床上传 Skill（可选）
- 缺失时显示安装指引或降级处理

### 第二步：识别内容来源

**四种内容类型：**

1. **对话总结**
   - 触发：用户说"总结一下"，且没有外部内容
   - 处理：回顾对话历史，提取核心观点

2. **文章链接**
   - 触发：输入包含 URL
   - 处理：使用 WebFetch 抓取内容，提取元数据

3. **截图/图片**
   - 触发：用户上传图片文件
   - 处理：使用 Claude 视觉分析，上传到图床

4. **纯文本**
   - 触发：用户直接粘贴文本
   - 处理：识别格式需求，直接保存或格式化

### 第三步：智能分类推荐

**推荐逻辑：**

1. 关键词匹配（精确匹配优先）
2. 关键词权重计算（高权重词影响大）
3. 上下文分析（结合对话历史）
4. 用户确认（显示推荐，允许修改）

**分类参考：**
- 从 `references/categories.md` 加载预定义分类
- 支持多级目录（如 `docs/backend/nodejs/express/`）
- 用户可以创建新分类

### 第四步：选择文档模板

**自动选择规则：**

- **精简版**：< 200 字
- **标准版**：200-1000 字
- **详细版**：> 1000 字 或 文章链接

**模板定义：**
- 从 `references/templates.md` 加载模板结构
- 包含 Front Matter（日期、分类、标签、来源等）
- 支持自定义模板（可扩展）

### 第五步：用户预览和确认

**显示内容：**
- 文件名（`[YYYY.MM.DD] 标题.md`）
- 分类目录
- 模板类型
- 完整文档预览

**用户选项：**
1. 直接保存并推送
2. 修改内容/标题
3. 更改分类目录
4. 切换模板类型
5. 取消操作

### 第六步：处理图片资源

**流程：**

1. 检测图片大小（< 500KB 直接上传，否则询问）
2. 调用图床 Skill 上传图片
3. 获取图床链接
4. 更新文档中的图片引用

**失败处理：**
- 上传失败：提示用户，移除图片引用，继续保存文档
- Skill 未安装：跳过图片上传，仅保存文本

### 第七步：推送到仓库

**本地备份：**
```bash
backup_dir="/tmp/knowledge-capture/$(date +%Y-%m-%d)"
mkdir -p "$backup_dir"
cp document.md "$backup_dir/[2026.03.17] 标题.md"
```

**Git 操作：**
```bash
cd $KNOWLEDGE_REPO

# 创建目录（如果不存在）
mkdir -p docs/category/

# 创建文件
cat > "docs/category/[2026.03.17] 标题.md" << 'EOF'
[文档内容]
EOF

# Git 提交
git add "docs/category/[2026.03.17] 标题.md"
git commit -m "docs: 添加 标题"
git push origin main
```

### 第八步：错误处理

**Push 失败：**
- 保留本地备份
- 提供 `/retry-push` 命令
- 显示手动推送指引

**图床上传失败：**
- 跳过图片，文档照常保存
- 提示用户稍后手动上传

**网络问题：**
- 引导用户检查连接
- 提供重试机制

**仓库不存在：**
- 询问用户仓库路径
- 保存配置到环境变量

## 🎨 设计亮点

### 1. 渐进式设计

- **基础功能**：对话总结、文本保存（核心）
- **增强功能**：文章抓取、图片上传（可选）
- **高级功能**：自定义模板、多仓库（扩展）

### 2. 友好的用户体验

- 清晰的格式和 emoji 反馈
- 主动询问而非猜测
- 多次确认关键操作
- 失败时提供清晰的解决方案

### 3. 灵活的分类体系

- 预定义常见分类
- 支持多级目录
- 允许用户自定义
- 智能推荐但不强制

### 4. 多模板支持

- 三种预定义模板
- 自动选择合适的模板
- 用户可以手动切换
- 支持自定义扩展

### 5. 可靠的备份机制

- 所有文档先保存到本地
- 失败时不丢失数据
- 提供重试命令
- 方便手动恢复

## 🚀 部署和使用

### 部署到 OpenClaw

1. **确保文件结构正确**：
   ```bash
   cd /Users/gui/Desktop/code/my-skills/skills/knowledge-capture
   ls -la
   # 应该看到：SKILL.md, README.md, references/ 等
   ```

2. **OpenClaw 配置**：
   - 将 `knowledge-capture/` 目录路径添加到 OpenClaw 的 skills 配置中
   - 或确保 OpenClaw 扫描 `/Users/gui/Desktop/code/my-skills/skills/` 目录

3. **首次使用**：
   ```
   用户：/summary
   AI：[触发 knowledge-capture skill]
   ```

### 配置环境变量（推荐）

```bash
# 添加到 ~/.zshrc 或 ~/.bashrc
export KNOWLEDGE_REPO=/path/to/your/knowledge-repo
export IMAGE_REPO=/path/to/your/image-repo  # 可选

# 重新加载
source ~/.zshrc
```

### 验证安装

```

```

## 🐛 已知限制

1. **WebFetch 限制**：某些网站可能有反爬机制，抓取可能失败
2. **图片大小限制**：建议 < 500KB，超大图片可能上传失败
3. **Git 操作依赖网络**：网络不稳定时推送可能失败
4. **对话历史长度**：Claude 上下文窗口有限，超长对话可能截断
